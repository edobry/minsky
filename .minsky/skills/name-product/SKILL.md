# name-product — generate and screen a name using the Lexicon framework

You are naming a new Minsky primitive — a new operand for the locked vocabulary, a sub-product, a microsite, a campaign, or a marketing surface — or you are evaluating a naming choice (ours or an adjacent product's). This skill is the **synthesis dual** of [`/analyze-adjacent-product`](../analyze-adjacent-product/SKILL.md): that skill _reads_ an existing name to extract what it constructs semiotically; this skill _generates_ a new name and screens it before it ships.

If you are reaching for the first descriptive label that "says what it does" (`task-runner`, `review-manager`), stop. Descriptive names forfeit both the cumulative and the asymmetric advantage (see `minsky-brand` §7.1, principle 1). The method below is the resource-constrained single-agent dual of Lexicon's full process — enough rigor to avoid the obvious failure modes without a 1,500-candidate agency engagement.

**Rename-neutral scope.** This skill encodes the naming _methodology_. It does NOT decide whether Minsky-the-head-noun should be renamed — that is the active **mt#2046** workstream's call (umbrella: "Brand-name exploration — should Minsky be renamed?"; position paper Draft at Notion `371937f03cb481e1983ded72834510ec`; memory `2021067b`). The trademark-collision screen below applies to _every_ candidate including the current name, but the keep/rename verdict is mt#2046's, not this skill's.

## When to invoke

- **New operand** for the locked vocabulary (`minsky-brand` §7) — a term that compounds with the referent (a new _Minsky <X>_).
- **New sub-product** that needs its own name.
- **New marketing surface** — microsite, campaign, launch handle (compose with [`/marketing-site-design`](../marketing-site-design/SKILL.md) §7a, the surface cut).
- **Evaluating an adjacent product's name** — when an `analyze-adjacent-product` read surfaces a naming question you want to score against the framework.
- **Sanity-checking a name already chosen** — run the polarization / compound / code-coherence / trademark checks retroactively; a name that fails them is a candidate for change.

## 1. The framework — three phases

Lexicon's process is **identify → invent → implement**. The single-agent form below compresses it but keeps the load-bearing moves.

### Phase 1 — Identify

1. **Ask the bidirectional-behavior question** (`minsky-brand` §7.1, principle 5): "How should the marketplace behave toward this thing, and how should it behave toward the marketplace?" This replaces the generic "what are its values / positioning / mission" framing and converges with Minsky's principal-substrate axis (mt#1931).
2. **Run the diamond exercise** (§3) to surface the message that becomes the name.
3. **Name the locked constraints.** For a Minsky primitive, the register is already locked: Cyberbrain / Section 9 (`minsky-brand` §2), the five-layer reference architecture (§4), and the existing operand set (Mesh / Cockpit / Section / Asks / Flock). The new name must cohere with that register, not invent a new one.

### Phase 2 — Invent

1. **Generate in generative mode, not evaluative mode.** Ask "what _could_ we do with this name?" — never "is this name good?" Humans default to noticing-what's-wrong; that reflex kills candidates before they're developed. Generate a wide set first; evaluate later.
2. **Run the synchronicity drill** (§4) to force adjacency to unrelated domains.
3. **Bias toward the construction rules** in `minsky-brand` §7.1:
   - **Sound-symbolism** — pull noise-generating phonemes for signal. The per-phoneme table is the single source of truth in [`/minsky-brand`](../minsky-brand/SKILL.md) §7.1 (V / Z / X / B / S / K / M / L-N) — consult it there; this skill does not duplicate it. Note the locked-vocab gap analysis: the `MINSKY` head-noun carries zero V/Z/X high-intensity phonemes, so high-intensity operands can compensate.
   - **Compound multiplier** — prefer a compound (_Minsky <operand>_) where 1 + 1 = 3; resist the client-reflex pull toward a single short token.
   - **Make intangible tangible** — for an abstract substrate, reach for a physical-operational noun (Mesh, Section, Cockpit, Flock), not a sophisticated coinage.
   - **Processing fluency** — if coining, build from familiar morphemes (Vercel = `ver-` + `-cel`), not abstract-from-engineering opacity (Codium).

### Phase 3 — Implement

Screen the surviving candidates through the four tests (§2) AND the mandatory trademark-collision screen (§5). Only a candidate that passes all of them advances to a naming decision. For a marketing surface, derive the visual treatment afterward per `marketing-site-design` §7 Step 5.

## 2. The screening checks

Run each finalist through all four. Any failure is disqualifying or a flag, not a footnote.

- **Polarization test.** Does the name provoke argument on first hearing? Comfortable team consensus is the warning sign — it means the name is too safe (Andy Grove's Pentium principle). Argument is energy.
- **Compound-multiplier check.** Does the name multiply associations (1 + 1 = 3), or sit as a lone token where a compound would carry two semantic circles? For operands, prefer the compound with the locked referent.
- **Processing-fluency check.** If coined, are the morphemes individually familiar? If a reader can't decode the parts, the whole reads as opaque.
- **Not-like-the-other-guys test.** Run the competitor drill (§3): would a buyer say "they're not like the other guys" _before_ knowing what the thing does? That predisposition-to-consider is the signal the name is working.

## 3. The diamond exercise

Four corners; iterate over days, not in an hour:

- **Win** — how do you define winning for this thing?
- **Have to win** — the assets already in hand (locked myth, product proof, the corpus).
- **Need to win** — the gaps; what's missing to win.
- **Have to say to win** — the message that closes the gap. **This message is what becomes the name.**

**The competitor drill** (used by the not-like-the-other-guys test): ask a buyer-archetype reader _"We just got a new competitor and their name is X — what do you think about them?"_ This surfaces what the name does for the imagination and bypasses the evaluative reflex.

## 4. The synchronicity drill (single-agent rotation)

Lexicon runs three disguised teams in parallel: one knows the real product, a second thinks it's naming (say) a bicycle, a third thinks it's naming an industrial system. The forced adjacency to unrelated domains produces words the on-the-nose team never reaches (naming a sailboat → read hunting and flying magazines).

The single-agent manual form is a **rotation**: deliberately generate three candidate sets under three disguised framings, then pool them.

1. **True framing** — name the thing as what it is (a Minsky cybernetic-substrate primitive).
2. **Disguised framing A** — pretend you are naming something physical and unrelated (a piece of climbing gear, a navigation instrument, a hand tool). Generate names for _that_.
3. **Disguised framing B** — pretend you are naming an industrial or institutional system (a rail network, a power grid, a survey instrument). Generate names for _that_.

Pool all three sets, then map the disguised candidates back onto the real thing. The cross-domain words are where the non-obvious, story-opening names live. Keep the framings genuinely separate while generating — collapsing them back to the true framing too early defeats the drill.

## 5. Prior-use / trademark-collision screen (MANDATORY)

**No candidate advances — including the current name — until it passes a prior-use screen.** A same-class registered mark is a disqualifier or a high-risk flag, not a footnote.

**Motivating finding (2026-05-31).** `MINSKY` is a **LIVE, INCONTESTABLE** registered mark in **IC 042**: dMetrics, Inc., **U.S. Reg. 5855970** (Principal Register, standard-character, Sec. 8 & 15 accepted 2026-05-29, senior — first use 2015), the AI/NLP platform behind their ~$100M DoD contract. The original Lexicon analysis missed this collision entirely; the keep/rename implications are mt#2046's to weigh. This screen exists so the same gap never recurs at _proposal_ time — every candidate gets screened before it advances. **(NEEDS VERIFICATION — out-of-repo: these registration details come from USPTO/TSDR and memory `2021067b`, last checked 2026-05-31. Trademark status is mutable; reconfirm via TSDR or counsel before relying on it for a naming decision.)**

**What to screen for.** Any registered or pending mark in the relevant classes:

- **IC 009** (downloadable / packaged software) and **IC 042** (SaaS / cloud / AI software services) are the load-bearing classes for a Minsky primitive. A LIVE registered mark in either, for related goods/services, is a high-risk flag.
- "Same class" is necessary but not sufficient — the real test is likelihood-of-confusion + relatedness of goods. Treat a same-class LIVE registration as a stop-and-assess, not an automatic kill; surface it to the principal with the relatedness read.

**Tooling (document the options; do NOT wire a tool as part of this skill's scope).**

1. **Markbase** — free, no-auth JSON API for fast risk-sizing: `GET https://api.markbase.co/search?q=<term>`. Daily USPTO mirror; returns word mark, owner, IC class, status code, and serial/registration number per hit. Use first to size the risk. _(Verified live 2026-05-31: HTTP 200, returns structured USPTO hits.)_
2. **USPTO TSDR API** — authoritative live-status confirmation by serial/registration number. Free key from `account.uspto.gov/api-manager`; or use trademark counsel for a reliance-grade clearance. Use to confirm a Markbase hit's current status (LIVE / DEAD / pending; Sec. 8 & 15 incontestability).
3. **Trademark-search MCP server** (`jordanburke/trademark-mcp-server`, MIT, on npm as `trademark-mcp-server`) — a **DEFERRED** option to wire only if naming work becomes recurring enough to justify an MCP tool. Not part of this skill's scope.

**Process.** (a) Markbase-search each finalist; (b) for any same-class hit, confirm live status via TSDR or counsel; (c) record owner, class, status, and serial/registration number in the naming write-up; (d) surface same-class LIVE registrations to the principal with the relatedness read before the candidate advances. Owning a domain does NOT make a name defensible — screen owned candidates too (the 2026-05-31 pass struck _Visor_ and _Agentic_ on trademark grounds despite domain availability; see memory `2021067b`).
