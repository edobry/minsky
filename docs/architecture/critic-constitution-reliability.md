# Critic Constitution — Reliability Guards

The reviewer service (`services/reviewer/`) posts one adversarial review per PR
event. The model output is generated by a reasoning-heavy LLM (currently
`openai:gpt-5`) whose output is mostly, but not always, a well-formed review.
This document records the post-generation guards that sit between the model
response and `submitReview` — boundary checks that catch known failure modes
before they reach GitHub.

The calibration concern (is the reviewer's rigor right for the PR?) lives in
mt#1110 and the prompt itself (`prompt.ts`). The reliability concern (is the
reviewer's output even well-formed?) lives here. They are independent — a
calibration change should not require touching this file, and vice versa.

## Guard 1 — Empty-output (mt#1125)

**Failure observed.** GPT-5 reasoning exhausts `max_completion_tokens` before
producing visible output. The provider returns `text === ""`. Without a guard,
the reviewer would post a review body consisting only of the header annotation
(`**Independent adversarial review (Chinese-wall)**` + reviewer line + tier) —
a visually "approved with no issues" review that is actually "model silently
failed."

**Guard.** `validateReviewOutput` in `review-worker.ts` checks
`output.text.trim().length > 0` immediately after `callReviewer`. On failure,
the reviewer posts a NEUTRAL COMMENT (`buildEmptyOutputSkipNotice`) explaining
the model returned no content, then returns `status="error"` so server logs
record it as a failure rather than a successful review.

**Observability.** The server logs a `review_result` line with
`status="error"` and a `reason` that includes the token breakdown
(prompt/completion/reasoning/total), so operators can see whether the model
ran out of output budget, ran out of reasoning budget, or something else.

## Guard 2 — Chain-of-thought leakage (mt#1212)

**Failure observed.** On PR #743 (2026-04-24) the reviewer posted a body whose
first few pages were raw model scratch:

> I will review AI types.
>
> Calling read_file on src/domain/ai/types.ts.
>
> Go.
>
> This time for sure.

… followed by hundreds of blank lines, and only then the actual `## Findings`
section. This is distinct from the empty-output class: the model produced
content, but part of it was internal reasoning that was supposed to stay
inside the scratchpad and leaked into the visible output. Observed once on
PR #743, then again in a different form on PR #753 (tool-loop fallback
narration: "I'll just proceed to next checks to save time"). The PR #753
calibration row in `project_mt1110_calibration_data.md` has the full
comparison.

**Guard.** `sanitizeReviewBody` in `sanitize.ts` detects the pattern by
scanning the prefix of the model output (everything before the first
recognisable heading) for three classes of signal:

1. **Blank-line run** — 20 or more consecutive newlines. No legitimate review
   body contains this; it is a high-confidence scratchpad-leakage signature.
2. **Strong scratch phrases** — `Calling <tool_name> on <path>.`, `Go.`
   on its own line, `This time for sure`, `[invoking]`, `Opening the file`,
   `Let's try again`, `tool call incoming`, `Sorry, executing now`,
   `I'll just proceed`, `to=functions.<tool>` (OpenAI tool-protocol routing
   leakage), and `tool glitch` / `tool is glitching` (tool-loop self-narration).
   Each single match is sufficient to fire the guard.
3. **Long narrative prefix** — `I will` / `I'll` / `I am going to` appearing
   within a prefix longer than 300 characters. A short "I will focus on the
   DI seam first…" sentence is legitimate intro prose and must not
   false-positive; the 300-char threshold is the narrative-tolerance cutoff.

**Decision: strip when a structural heading follows, error-out when it does
not.** If the model eventually produced a recognisable heading (`Findings`,
`Spec verification`, `Summary`, `Documentation impact` — as any of `#`
through `######` or `**Bold**` — case-insensitive), the guard strips the
leaked prefix and keeps the review body intact. If no such heading exists,
the leak is treated as a worker-side failure: the review body is replaced
with a structured `reviewer-service error: chain-of-thought leakage detected`
notice, the event is forced to COMMENT, and the result returns
`status="error"`.

Rationale for the two-action design: the stripped case preserves the
reviewer's actual signal (real findings with real evidence) and discards
only the leaked prefix; the error case avoids posting garbage when no
signal is recoverable. A single "always error-out" design would lose real
review content on the more common failure mode (leak-then-review); a
single "always strip" design would post whatever came after the leak
even if it were also garbage.

**Implementation notes.** Apostrophe character classes include both the
ASCII `'` and the typographic curly `’` — GPT-5 frequently emits curly
apostrophes in prose, so scratch phrases like `I'll`, `I'll`, `Let's`,
`Let's` all match. The blank-line-run regex accepts both LF and CRLF line
endings. The `Calling <tool>` narration pattern requires either a
snake_case tool name or an `on <path>` segment, avoiding false positives
on ordinary prose like "Calling maintainers,".

**Observability.** When the guard fires, `review-worker.ts` logs a
structured event `reviewer.cot_leak_detected` with:

- `prUrl` — the canonical PR URL for cross-referencing
- `commitSha` — the head SHA the review was on
- `originalLength` / `cleanedLength` — how much content was removed
- `action` — `stripped` or `errored`
- `reason` — joined signal list, e.g.
  `cot-leak:blank-line-run,scratch:this-time-for-sure,scratch:go-dot`
- `provider` / `model` — which model family exhibited the leak, so
  cross-provider comparisons are possible if/when we route to a
  non-OpenAI reviewer

These logs feed the mt#1110 calibration data file when a new leak class
appears.

## Non-guards (out of scope for this file)

- Prompt-level changes to the Critic Constitution itself → mt#1110.
- Model-internal reasoning-token handling → upstream provider.
- Tool-call scaffolding that correlates with the tool-loop-fallback
  sub-pattern → mt#1126 (tool wiring) and mt#1189 (cross-model
  convergence) are the relevant tickets.
