# Workspace-COPY Pre-Commit Guard

> Extracted from `.minsky/rules/hook-files.mdc` (mt#2620) — full incident narration,
> cross-references, and worked examples for this hook/guard. The compiled rule corpus
> carries only a terse index entry; this file is the durable detail.

**Formerly a detect-and-block guard; now a generate-and-restage step
(mt#2621).** A step in the pre-commit pipeline (`src/hooks/pre-commit.ts`,
between the NUL-byte check and the TypeScript type check) **regenerates**
the workspace package.json COPY block in every Dockerfile in the repo that
runs `bun install --frozen-lockfile` against the root `bun.lock` (root +
`services/*/Dockerfile` + `packages/*/Dockerfile`) from root
`package.json`'s `workspaces` glob, and re-stages any Dockerfile that
changed. This mirrors the mt#2622 completion-manifest auto-fix-and-restage
pattern (Step 1b) rather than the detect-and-block shape the guard used
from mt#1984 through mt#2607: instead of catching drift between the glob
and a hand-typed COPY list after the fact, the COPY lines are never
hand-typed at all after the one-time marker setup, so the drift class the
original guard patrolled cannot recur.

The discovery is by trigger condition (has `bun install --frozen-lockfile`),
not by hardcoded filename. It applies to the _class_ of "Dockerfiles that
install against the root bun.lock" rather than to any specific instance;
sub-project Dockerfiles that adopt frozen-lockfile installation are
automatically protected without code changes (mt#1992) — currently root
`Dockerfile`, `services/reviewer/Dockerfile`, and
`services/cockpit/Dockerfile` (the latter a two-stage build with two
marker-delimited blocks, one per stage).

**Hook file (in-pipeline step):** `src/hooks/pre-commit.ts` →
`runDockerfileWorkspaceCopyRegen()`, which shells out to
`bun run generate:dockerfile-workspace-copies`
(`scripts/generate-dockerfile-workspace-copies.ts`) and then diffs/stages
any changed Dockerfile via `git diff --name-only` / `git add`. Pure-function
implementation: `src/hooks/workspace-copy-detector.ts` (key exports:
`discoverProtectedDockerfiles`, `resolveWorkspacePackageJsonPaths`,
`renderWorkspaceCopyBlock`, `applyGeneratedWorkspaceCopyBlock`,
`planDockerfileWorkspaceCopyRegeneration`).

**How generation works.** Each protected Dockerfile contains a
marker-delimited block:

```dockerfile
# === BEGIN GENERATED: workspace package.json COPY (mt#2621) ===
# Generated by `bun run generate:dockerfile-workspace-copies` ...
COPY packages/domain/package.json ./packages/domain/package.json
COPY packages/shared/package.json ./packages/shared/package.json
...
# === END GENERATED: workspace package.json COPY ===
```

`applyGeneratedWorkspaceCopyBlock` replaces EVERY marker-delimited span in
the file (handling multi-stage Dockerfiles with more than one block) with
freshly rendered `COPY <ws>/package.json ./<ws>/package.json` lines, sorted
by workspace path. A Dockerfile that runs the frozen-lockfile install but
has never adopted the markers fails the regeneration step with an error
naming the missing markers — a one-time setup gap, not staleness — rather
than silently skipping it.

**Why this guard existed / what it caught (history).**

_Originating incident (mt#1977 / 2026-05-20T19:44Z)._ PR #1186
(mt#1934, marketing-site rebuild) added `services/site/` as a new
workspace declared by root `package.json`'s
`workspaces: ["packages/*", "services/*"]`. The regenerated `bun.lock`
referenced `services/site`'s deps. The root Dockerfile's selective
workspace-COPY block only copied `packages/shared/package.json` and
`services/reviewer/package.json` — `services/site/package.json` was
missed. Every Railway deploy from `57c2e868` onward failed at
`RUN bun install --frozen-lockfile` with
`error: lockfile had changes, but lockfile is frozen`. Production was
stuck on the prior commit for ~75 minutes.

_Generalization incident (mt#1991 / 2026-05-20T20:55Z, ~6 hours later)._
The mt#1984 guard's original scope was root-Dockerfile-only. The same
failure class hit `services/reviewer/Dockerfile` (a sub-project Dockerfile
with its own `bun install --frozen-lockfile` step against the root
lockfile). After mt#1934 added services/site as a workspace + mt#1983
added winston, the reviewer Dockerfile's COPY block silently fell out
of date: 8 consecutive FAILED Railway deploys over ~4 hours, until
mt#1989 added a `deploy.config.ts` that gave the deployment_status MCP
tool visibility into the reviewer service. mt#1991 hotfixed the
Dockerfile; mt#1992 generalized the (then detect-and-block) guard to walk
all protected Dockerfiles, not just root.

_Retirement of the detect-and-block shape (mt#2621 / mt#2607)._ A July
2026 audit (mt#2607, services/infra finding 12) named the pattern
explicitly: "the guards defend the fragility; remove the fragility
instead." Two production outages had already been caught by the
detect-and-block guard; mt#2621 removed the drift class it was patrolling
by generating the COPY block instead of hand-maintaining it.

**Trigger condition.** Always — the step runs on every commit regardless
of which files are staged, mirroring Step 1b's completion-manifest
rationale: the generator is a pure function of root `package.json`'s
`workspaces` field and completes in well under a second, so there is no
benefit to gating on which files are staged.

**On a regeneration failure:** the step blocks with the generator's error
output — this happens only when a protected Dockerfile is missing the
generated-block markers (see "How generation works" above), not on
ordinary drift (which the generator fixes automatically rather than
blocking on).

**Allowlist (excluded from the protected set):**

- Workspaces matched by the glob but lacking a `package.json` — these
  aren't workspaces from bun's perspective (the glob walks for
  `package.json`), so no COPY is generated for them. The mt#1977 instance
  of this is `services/minsky-mcp/`, a directory that matches
  `services/*` but exists only as a service-config home (no package.json).
- Dockerfiles with no `RUN bun install --frozen-lockfile` line — they
  don't install against the root lockfile, so the workspace-COPY
  invariant doesn't apply to them. Such files are discovered, evaluated
  against the trigger condition, and dropped from the protected set.
- Repositories without a root `package.json` — silent short-circuit (the
  generator needs the root workspaces field to determine what to
  generate; absent that there's no contract to satisfy).
- Repositories with a root `package.json` but NO protected Dockerfiles —
  silent pass.

**No override.** Unlike the retired detect-and-block guard (which had
`MINSKY_SKIP_WORKSPACE_COPY_CHECK`), this step has no skip mechanism — like
Step 1b's completion-manifest regen, it only ever auto-fixes forward, so
there's nothing to override. `MINSKY_SKIP_WORKSPACE_COPY_CHECK` and its
`WORKSPACE_COPY_CHECK_OVERRIDE_ENV` constant were removed from
`src/hooks/workspace-copy-detector.ts` and from `HOOK_ONLY_ENV_VARS` at
`packages/domain/src/configuration/sources/environment.ts` in mt#2621.

**Glob-expander support.** The resolver currently handles two glob forms:
literal workspace paths (`packages/shared`) and single-`*` trailing globs
(`packages/*`, `services/*`). Patterns with `**`, negations
(`!packages/*-archived`), or character classes (`[abc]*`) are SKIPPED
rather than mis-interpreted — the resolver is conservative and would
prefer to under-generate than mis-generate. If Minsky's root
`package.json` ever adopts those patterns the resolver will need
extension; a follow-up task should be filed at that time. The current
patterns (`packages/*`, `services/*`) are fully covered.

**Performance:** regeneration completes in well under a second in
practice (a handful of fs reads + string templating + one subprocess
invocation). Scales linearly with the number of protected Dockerfiles;
current count is 3 (root, services/reviewer, services/cockpit).
